理解Docker容器网络

本节概述了Docker引擎原生提供的默认网络特性，里面介绍了Docker默认创建的网络类型，以及如何创建用户定义的网络。此外还介绍了在单个主机或集群上创建网络所需的资源。

默认网络

Docker在安装时自动创建了三个网络。您可以通过运行docker network ls命令列出这些网络：

$ docker network ls

NETWORK ID          NAME                DRIVER
7fca4eb8c647        bridge              bridge
9f904ee27bf5        none                null
cf03ee007fb4        host                host

Docker历来的实现中包含了这三个网络类型。当您运行一个容器的时，您可以通过--network标识来指定容器运行在哪个网络中。

名称为bridge的网络实际上就是Docker安装的docker0网络，除非通过docker run --network=<NETWORK>选项指定了网络，否则默认情况下Docker守护进程把容器都连接到这个网络。您可以通过在宿主机上运行ifconfig命令来查看，可以看到这个bridge网络是宿主机网络栈的一部分。

$ ifconfig

docker0   Link encap:Ethernet  HWaddr 02:42:47:bc:3a:eb
          inet addr:172.17.0.1  Bcast:0.0.0.0  Mask:255.255.0.0
          inet6 addr: fe80::42:47ff:febc:3aeb/64 Scope:Link
          UP BROADCAST RUNNING MULTICAST  MTU:9001  Metric:1
          RX packets:17 errors:0 dropped:0 overruns:0 frame:0
          TX packets:8 errors:0 dropped:0 overruns:0 carrier:0
          collisions:0 txqueuelen:0
          RX bytes:1100 (1.1 KB)  TX bytes:648 (648.0 B)

您可以通过CTRL-p CTRL-q组合按键从容器中退出并保持容器运行。

名称为host的网络把容器添加到宿主机的网络栈中，您可以在容器内部查看网络栈信息，您会发现和宿主机的信息是一模一样的。

除了bridge网络，您一般不需要与这些默认网络进行交互。您可以列出并检视这些网络，但不能删除它们，它们是Docker必须安装的。然而，您可以创建您自有的用户定义型的网络，当然您也可以在不需要的时候删除掉。在您了解更多有关创建自有网络之前，值得再深入看一下默认的bridge网络。

默认的docker0网络支持端口映射，docker run --link命令可以使docker0网络里的容器互相通信，这些技术手段设置繁琐且易于出错，虽然仍然可用，但最好避免使用并使用自定义的bridge网络来替代。

用户定义型网络

您可以创建您自有的用户定义型网络来实现更好的容器隔离性。Docker提供了一些默认的网络驱动用以创建自定义网络。您可以创建一个新的bridge网络，overlay网络或者MACVLAN网络，您也可以根据自己的需求编写一个网络插件（network plugin）或远程网络（remote network）。

您可以创建多个网络，您也可以把多个容器加到多个网络中。容器可以在其所属的网络中互相通信，但不能跨越其不所在的网络。一个连接到两个网络的容器可以与这两个网络中的容器互相通信。当一个容器连接到了多个网络中时，其出站连接是由第一个（按词汇文本顺序排序）非内部网络提供。

下面的几节详细描述了每个Docker自带的网络驱动。

bridge网络

最容易创建的用户定义型网络是bridge网络。这个网络与默认的docker0网络类似，但有些额外的功能，也有些缺失的功能。

$ docker network create --driver bridge isolated_nw
1196a4c5af43a21ae38ef34515b6af19236a3fc48122cf585e3f3054d509679b

$ docker network inspect isolated_nw

            "Driver": "default",

                    "Subnet": "172.21.0.0/16",

只有在与自定义网络相同的Docker宿主机上启动的容器才能连接到这个自定义网络，这个网络里的容器可以立即与其它容器进行通信。然而，这个网络本身把内部的容器与外部的网络隔离开了。

在一个用户定义型的网络中是不支持连接（linking）的。您可以在网络中的容器上公开端口，以便向外部网络公开bridge网络内的部分内容。

对于在单一主机上规划一个相对小的网络，bridge网络比较有用。而对于大型网络，您可以通过创建一个overlay网络。

Docker引擎swarm模式的overlay网络

在没有外部key-value存储的情况下，您可以在一个以swarm模式运行的管理节点上创建overlay网络。swarm仅把overlay网络提供给swarm集群里的节点。当您创建一个需要使用overlay网络的服务时，管理节点自动在服务节点上添加这个overlay网络。

更多关于Docker引擎swarm模式的信息，请参考swarm模式简介

下面的例子展示了如何创建一个overlay网络，并把网络提供给swarm集群里的服务节点。

# Create an overlay network `my-multi-host-network`.
$ docker network create \
  --driver overlay \
  --subnet 10.0.9.0/24 \
  my-multi-host-network

400g6bwzd68jizzdx5pgyoe95

# Create an nginx service and extend the my-multi-host-network to nodes where
# the service's tasks run.
$ docker service create --replicas 2 --network my-multi-host-network --name my-web nginx

716thylsndqma81j6kkkb5aus

swarm使用的overlay网络不能供由docker run命令创建的非swarm模式的容器使用。更多信息请参考Docker swarm模式下overlay网络的安全模型。

也请参考连接服务至overlay网络。

使用外部key-value存储的overlay网络

假如您不使用Docker引擎的swarm模式而创建overlay网络，您需要一个额外可用的key-value存储服务。支持的key-value存储服务包括Consul，Etcd和Zookeeper。在创建一个overlay网络之前，您需要安装配置好一个选定的key-value存储服务。规划网络中的Docker宿主机必须能够与这个key-value存储服务进行通信。

注意：Docker引擎swarm模式与使用外部key-value存储的网络不兼容。

网络中的每个宿主机都必须运行着Docker引擎，最简单的方式是使用Docker Machine来提供这些宿主机。

您需要在每个宿主机上开启下面的端口。

Protocol Port Description
udp 4789 Data plane (VXLAN)
tcp/udp 7946 Control plane

外部的key-value存储服务也许需要额外的端口。查看官方的文档并开启其它需要的端口。

一旦您提供了几台宿主机，您可以使用Docker Swarm来快速把它们组成一个具有服务发现的swarm集群。

为了创建overlay网络，您需要在每个Docker引擎的守护进程里配置下面的三个选项：

Option Description
--cluster-store=PROVIDER://URL Describes the location of the KV service.
`–cluster-advertise=HOST_IP\ HOST_IFACE:PORT` The IP address or interface of the HOST used for clustering.
--cluster-store-opt=KEY-VALUE OPTIONS Options such as TLS certificate or tuning discovery Timers

在swarm集群里的其中一台宿主机上创建一个overlay网络。

$ docker network create --driver overlay my-multi-host-network

这样就创建了一个跨多主机的网络，overlay网络提供了容器的完全隔离。

然后在每台宿主机上，指定网络的名称来启动容器。

$ docker run -itd --network=my-multi-host-network busybox

一旦连接后，不管容器是在哪台宿主机上，网络里所有的容器都可以互连。

自定义网络插件

如果需要的话，您也可以创建自己的网络驱动插件。网络驱动插件利用Docker的插件机制，这种机制下，插件是一个运行在与Docker守护进程相同宿主机上的进程。

网络插件与其它插件一样遵循同样的限制和安装规则。所有的插件都调用插件API，它们都需要经历安装、启动、停止和激活的生命周期。

一旦您创建也安装了一个自定义网络插件，您可以像使用自带的网络驱动一样使用它，如：

$ docker network create --driver weave mynet

您可以检视这个网络信息，添加容器到这个网络中等。当然不同的插件也许会使用不同的技术或框架。自定义网络也能包含Docker默认网络不具备的功能。有关编写插件的更多信息，请参考扩展Docker和编写网络驱动插件。

Docker内嵌的DNS服务

Docker守护进程内嵌了一个DNS服务用于连接到自定义网络中的容器的自动服务发现。容器名的解析首先由这个内嵌的DNS服务器处理。假如内嵌的DNS服务无法解析，解析请求会转发给为容器配置的DNS服务器。为此在创建容器时，只有内嵌的DNS服务器可以通过127.0.0.11访问到，这个配置在容器的resolv.conf文件中。更多有关自定义网络的内嵌DNS服务器信息，请参考Embedded DNS server in user-defined networks。

连接（Links）

在有Docker的网络特性之前，您可以使用Docker的连接（Link）来实现容器间的互发现。引入Docker网络后，容器可以通过其名字自动发现。但您仍然可以使用连接，但用户定义型网络下的连接工作方式会与默认的docker0网络下的连接工作方式有所不同。有关默认docker0桥接网络下的连接特性，请参考Legacy Links，有关用户定义型网络下的连接特性，请参考linking containers in user-defined networks。
